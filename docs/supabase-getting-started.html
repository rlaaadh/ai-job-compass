Supabase Setup Guide

현재 프로젝트에서 Supabase 시작하는 방법

이 문서는 현재 이직각 프로젝트를 기준으로, 로컬 개발과 Vercel 배포를 모두 고려했을 때 Supabase를 어떻게 도입하면 좋은지 설명하는 시작 가이드입니다. 목표는 "회사 검색용 DB와 사용자 데이터 저장소를 Supabase(Postgres)로 옮길 준비"를 하는 것입니다.

왜 Supabase인가

Vercel과 궁합이 좋다

SQLite는 로컬 파일 기반이라 서버리스 배포와 잘 맞지 않지만, Supabase는 원격 Postgres라 배포에 유리합니다.

Postgres 기반이다

현재 설계한 검색용 DB를 SQLite 개념 그대로 Postgres로 확장하기 쉽습니다.

추후 확장에 유리하다

회사 검색 외에도 사용자 프로필, 즐겨찾기, 비교 기록, 인증까지 확장할 수 있습니다.

이 프로젝트에서 Supabase를 어디에 쓸까

1차 도입 범위

  • 회사 검색용 DB
  • 월별 통계 저장
  • 검색 alias / normalized_name 저장
  • 향후 배치 적재 대상 저장소
가장 먼저 추천 현재 요구와 직결

2차 도입 범위

  • 사용자 프로필 저장
  • 비교 결과 히스토리
  • 인증(Auth)
  • RLS 정책 기반 사용자별 데이터 보호
나중 단계 지금은 없어도 됨

시작 순서
1

Supabase 프로젝트 생성

새 프로젝트를 만들고 DB 비밀번호를 설정합니다.

2

연결 정보 확인

Dashboard의 Connect 화면에서 DB 연결 문자열을 확인합니다.

3

환경변수 저장

로컬과 Vercel에 DB URL, Supabase URL, anon key 등을 저장합니다.

4

스키마 생성

companies, monthly_stats, aliases 같은 검색용 테이블을 생성합니다.

5

백엔드 연결

FastAPI가 SQLite 대신 Supabase Postgres를 읽도록 연결합니다.

6

검색 API 교체

/companies/search 가 공공 API 대신 Supabase DB를 조회하도록 전환합니다.

환경변수 설계
변수명 어디서 사용 설명
SUPABASE_URL 프론트/백엔드 Supabase 프로젝트 URL
SUPABASE_ANON_KEY 프론트 공개 가능한 anon key. 클라이언트 SDK에서 사용 가능
SUPABASE_SERVICE_ROLE_KEY 백엔드 절대 프론트에 노출하면 안 되는 서버용 키

SUPABASE_ANON_KEY=your_anon_key
SUPABASE_SERVICE_ROLE_KEY=your_service_role_key
SUPABASE_DB_URL=postgresql://postgres:[PASSWORD]@aws-[REGION].pooler.supabase.com:6543/postgres

현재 프로젝트에서 추천 연결 방식

백엔드(FastAPI) → Supabase DB

  • 검색용 DB 조회와 배치 적재는 FastAPI가 직접 Postgres에 연결하는 방식이 가장 단순합니다.
  • 즉, 지금의 SQLAlchemy 계층을 유지하면서 DB 엔진만 SQLite에서 Postgres로 바꿉니다.
  • 검색/스코어 계산은 백엔드에서 계속 담당합니다.

프론트(Next.js) → Supabase

  • 지금 단계에서는 프론트가 Supabase DB를 직접 읽지 않아도 됩니다.
  • 검색은 기존처럼 백엔드 API를 호출하는 편이 구조가 더 깔끔합니다.
  • 추후 로그인/프로필 저장이 필요해지면 그때 Supabase JS SDK를 붙이면 됩니다.

테이블 지금 필요한가 용도

개발 단계별 로드맵

1단계: DB 연결 성공

Supabase 프로젝트 생성, connection string 저장, FastAPI에서 접속 확인

2단계: 검색용 스키마 생성

companies / aliases / monthly_stats 테이블 생성

3단계: 적재 스크립트 작성

국민연금 API에서 데이터를 읽어 Supabase에 넣는 배치 스크립트 작성

4단계: 검색 API 전환

/companies/search 를 Supabase 조회 기반으로 변경

5단계: 상세/비교도 DB 우선화

건강도 계산도 공공 API 직조회보다 DB 저장 데이터를 우선 사용

6단계: Vercel 배포

로컬 .env 값을 Vercel 환경변수로 옮겨 배포 환경 구성

Vercel 배포 관점 팁

중요 포인트

  • 프론트가 Vercel에서 뜨더라도 DB는 Supabase 원격 Postgres를 사용하면 됩니다.
  • 백엔드도 Vercel이 아니라 별도 서버/플랫폼에 둘 수 있고, 이후 통합 여부를 결정해도 됩니다.
  • 서버리스성 트래픽은 풀러(connection pooler) 연결 문자열을 고려해야 합니다.

실무적으로는

  • 프론트: Vercel
  • DB: Supabase
  • 백엔드: 초기엔 로컬/별도 API 서버, 이후 필요시 서버리스/컨테이너 이전
RLS는 언제 고려하나
Supabase 공식 가이드 기준으로, 프론트에서 Data API를 직접 사용할 때는 RLS(Row Level Security)가 중요합니다. 다만 현재 프로젝트는 검색과 계산을 FastAPI 백엔드가 담당하므로, 1차 도입 단계에서는 "서버에서 DB를 읽는 구조"를 먼저 완성하고, 그 다음 사용자 데이터가 생길 때 RLS를 본격 적용하는 순서를 추천합니다.
현재 프로젝트 기준 최적의 시작점

지금 바로 할 것

  • Supabase 프로젝트 생성
  • DB 연결 환경변수 추가
  • 검색용 스키마를 Postgres 기준으로 준비
  • 기존 SQLite 문서를 Supabase/Postgres 기준 구현으로 옮기기

아직 안 해도 되는 것

  • 프론트 직접 Supabase 연결
  • Auth 붙이기
  • Realtime
  • Edge Functions
한 줄 결론
현재 이직각 프로젝트에서 Supabase를 시작하는 가장 좋은 방법은 "프론트부터 Supabase에 직접 붙이는 것"이 아니라, 먼저 FastAPI 백엔드의 검색용 DB를 Supabase Postgres로 바꾸는 것입니다.
