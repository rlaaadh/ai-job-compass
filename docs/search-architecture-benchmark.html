Search Architecture Benchmark

현재 구조 vs 좋소판별기 스타일 구조

이 문서는 현재 이직각 프로젝트의 검색 구조와, 벤치마킹 대상처럼 "타이핑할 때 바로바로 결과가 뜨는 구조"의 차이를 설명하는 기획 문서입니다. 핵심은 단순히 프론트 UI 문제가 아니라, 검색 데이터를 어디서 가져오고 어떤 방식으로 준비해두느냐의 차이입니다.

한 줄 요약
지금 프로젝트는 사용자가 검색할 때마다 공공 API에 바로 물어보는 구조이고, 벤치마크 서비스 같은 빠른 검색 UX는 보통 미리 저장해둔 자체 데이터베이스나 검색 인덱스를 조회하는 구조입니다.
구조 비교

현재 이직각 구조

실시간 공공 API 의존형

  • 사용자가 회사명을 입력함
  • 프론트가 백엔드 /companies/search 호출
  • 백엔드가 국민연금 API를 실시간 호출
  • 응답을 정리해서 프론트에 반환
최신 원본 데이터 느릴 수 있음 간헐적 503 가능

벤치마크형 구조

미리 적재된 검색 DB 조회형

  • 사용자가 회사명을 입력함
  • 프론트가 검색 API 호출
  • 백엔드가 자체 DB/검색 인덱스를 즉시 조회
  • 빠른 자동완성 결과를 바로 반환
빠른 응답 안정적 UX 배치 적재 필요
왜 체감 속도가 차이나는가

1. 데이터 준비 상태

현재 구조는 검색할 때마다 원격 서버에서 새로 데이터를 받아옵니다. 반면 벤치마크형은 이미 저장된 데이터를 바로 조회합니다.

2. 외부 API 의존성

공공 API는 느리거나 불안정할 수 있습니다. 자체 DB 검색은 우리 서비스 안에서 끝나기 때문에 훨씬 예측 가능합니다.

3. 검색 최적화 가능 여부

자체 DB는 인덱스, LIKE 검색, 전문 검색, 추천 검색어 같은 최적화가 가능하지만, 공공 API는 제공되는 파라미터 범위 안에서만 써야 합니다.

2

6

드롭다운 옵션 표시

벤치마크형 구조의 흐름
1

배치 수집

백엔드가 미리 국민연금 데이터를 수집

2

DB 저장

회사 검색용 테이블에 적재

3

인덱스 생성

회사명 검색이 빠르도록 준비

4

사용자 입력

예: 풀무원 입력

5

즉시 DB 조회

외부 API가 아니라 내부 DB에서 검색

6

빠른 결과 반환

자동완성 UX가 안정적이고 빠름

현재 구조의 장단점

장점

  • 초기 구현이 단순합니다.
  • 원본 공공 데이터를 바로 조회하므로 별도 적재 작업이 적습니다.
  • MVP 단계에서 "일단 동작하는 검색"을 빠르게 만들 수 있습니다.

단점

  • 검색 속도가 외부 공공 API 상태에 직접 영향을 받습니다.
  • 자동완성처럼 연속 요청이 많은 UI와 궁합이 좋지 않습니다.
  • 503, timeout, socket hang up 같은 예외가 UX에 바로 드러날 수 있습니다.
벤치마크형 구조의 장단점

장점

  • 검색 응답이 빠르고 예측 가능합니다.
  • 자동완성 UX가 훨씬 자연스럽습니다.
  • 검색 품질 개선(정렬, 추천, 인기순, 오타 보정)이 쉽습니다.

단점

  • 배치 수집/동기화 로직을 추가로 만들어야 합니다.
  • DB 설계와 업데이트 전략이 필요합니다.
  • "최신 원본 데이터"와 "우리 DB 반영 시점" 사이에 시간차가 생깁니다.
기획 관점에서의 차이
항목 현재 구조 벤치마크형 구조
검색 데이터 원천 검색 시점의 국민연금 API 미리 적재한 자체 DB/검색 인덱스
응답 속도 외부 API 상태에 따라 달라짐 대체로 빠르고 일정함
안정성 503/끊김 가능 상대적으로 높음
자동완성 적합성 낮음 높음
초기 구현 난이도 낮음 중간 이상
확장성 제한적 높음
이직각이 벤치마크형으로 가려면

1단계: 회사 검색용 DB 만들기

회사명, 산업명, 직원 수, 최신 월 데이터 등을 저장하는 검색용 테이블을 구성합니다.

2단계: 배치 수집 추가

국민연금 API를 사용자 요청 때마다 부르지 않고, 정해진 주기로 수집해서 DB를 업데이트합니다.

3단계: 검색 API를 DB 기반으로 교체

/companies/search 가 더 이상 외부 API를 직접 치지 않고 DB를 검색하게 바꿉니다.

4단계: 상세 분석만 원본/캐시 활용

자동완성은 DB로 빠르게 처리하고, 더 깊은 분석이 필요할 때만 추가 데이터를 불러오는 식으로 분리할 수 있습니다.

5단계: 정렬 전략 강화

정확도순, 직원 수 기준, 브랜드 인지도, 최근성 같은 정렬 기준을 설계할 수 있습니다.

6단계: UX 최적화

검색 취소, 이전 요청 무시, 캐시, 추천 검색어, 최근 검색어 같은 UX가 더 자연스럽게 작동합니다.

추천 방향

지금 당장 할 일

  • 현재 구조는 유지하되, 짧은 검색 차단과 캐시를 더 보강합니다.
  • 메인 검색 UX가 깨지지 않게 에러 메시지를 더 부드럽게 다듬습니다.
  • MVP 시연용으로는 충분히 사용할 수 있게 안정화에 집중합니다.

다음 버전에서 할 일

  • 검색은 자체 DB 기반으로 전환합니다.
  • 공공 API는 배치 수집/업데이트 용도로 격리합니다.
  • 벤치마크 서비스 같은 체감 속도는 이 단계에서 비로소 가능해집니다.
핵심 결론
벤치마크 서비스처럼 "타이핑하자마자 바로 뜨는 검색"을 만들고 싶다면, 프론트만 다듬는 것으로는 부족합니다. 검색용 데이터를 우리 서버 안에 미리 저장해 두는 구조 변화가 필요합니다.
